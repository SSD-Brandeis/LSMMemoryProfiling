\subsection{Experimental Setup}
To conduct these experiments, we used a Chameleon machine equipped with a
configuration of 187 GiB of RAM and 48 cores. We used the
RocksDB~\cite{Facebook2021} storage engine to evaluate the performance of the
write buffer implementations. The RocksDB was configured to use only one thread
for compactions and one thread for flushes to ensure that the performance is
not affected by the compaction and flush threads. We also disabled the
write-ahead log (WAL) to ensure that the performance is not affected by the
WAL. Below are the parameters that we use across all the experiments:
\begin{itemize}[leftmargin=*,labelindent=0mm, itemsep=0.2\baselineskip]
    \item \textit{E} (entry size) = 64 bytes
    \item \textit{B} (number of entries that fit into a page) = 64
    \item \textit{P} (size of buffer in pages) = 4096
    \item \textit{T} (size ratio between adjacent levels) = 4
    \item \textit{M} (main memory allocated to write buffer) = 16 MB
\end{itemize}

\subsection{Memory Footprint}
The memory requirements of various write buffer structures vary when tasked
with storing the same volume of data. For instance, a \textit{skip list}
generally requires more memory than a \textit{vector} due to its layered
structure and the necessity to maintain pointers. Similarly, a \textit{hash
    skip list} has a higher memory footprint than a \textit{skip list} because of
the additional overhead from a hash table and pointers. This experiment seeks
to reveal the trade-offs between the memory footprint and the storage capacity
(number of entries) of each write buffer implementation, providing insights
into their efficiency and appropriateness for different scenarios.

We conducted the experiment by inserting 1 million entries into each write
buffer implementation and measured the memory footprint. The bucket count
(\textit{H}) and the prefix length (\textit{X}) for the \textit{hash skip list}
and \textit{hash linked list} were set to 1 and 0, respectively.
Figure~\ref{fig:memory_footprint} demonstrates that the \textit{vector}
exhibits the lowest memory footprint due to its contiguous storage and absence
of pointers, whereas the \textit{hash skip list} displays the highest memory
footprint, reflecting its larger memory requirement. The \textit{skip list} and
\textit{hash linked list} have memory footprints that lie between the
\textit{vector} and the \textit{hash skip list}.

\subsection{Read Performance}

\Paragraph{Point Queries} The performance of point queries is a critical factor in the selection of a write buffer
implementation. In the case of a \textit{vector}, point queries exhibit an $O(n)$ complexity, as the write buffer must be
scanned linearly to locate the required key. In contrast, \textit{skip list} implementations
offer improved read performance, with point queries achieving $O(\log n)$ complexity. The  \textit{hash skip list} and \textit{hash skip list}, in
particular, can deliver the best performance for point queries if the prefix length and the number of buckets are
optimally tuned. It is noteworthy that the performance varies based on configuration,
and it falls between \textit{vector} and \textit{skip list}.

The prefix length (\textit{X}) and the number of buckets (\textit{H}) can
fairly influence the performance of the \textit{hash skip list} and the
\textit{hash linked list}. There are two extreme cases: $\left. 1\right)$
setting the prefix length too small, for example, to 0, results in all keys
being stored in a single bucket, $\left. 2\right)$ increasing the prefix length
to the maximum length of the keys while maintaining only one bucket leads to
the same bottleneck. Ideally, the number of buckets should approximate
$256^{\textit{X}}$ (assuming the character based prefix) to optimize
performance. This strategy, however, poses a challenge as it necessitates a
substantial increase in the number of buckets—approximately 65K for a
\textit{X} of 2 and about 16M for a \textit{X} of 3—thereby significantly
raising the memory overhead due to the need for each bucket to maintain a
pointer to a \textit{skip list}, unless it is empty. For point queries,
achieving an $O(1)$ complexity is possible when the \textit{X} is adjusted to
the maximum key length, coupled with $256^{\textit{X}}$ buckets, provided that
sufficient memory is available. Moreover, the efficiency of point queries is
further optimized when the distribution of keys across buckets is uniform.
Thus, the \textit{hash skip list} can attain optimal performance for point
queries through careful adjustment of the prefix length and the number of
buckets.

The inherent increase in memory attributes to the layered structure of the
\textit{hash skip list}. The \textit{hash linked list} emerges as a more
memory-efficient alternative. Despite this, point queries can still maintain
commendable performance provided that the buckets are uniformly filled and
contains a smaller number of entries.

To assess the performance of point queries on write buffers, we conducted an
experiment in which we inserted 140K entries, each 64 bytes in size, with
random interleaved 200 point queries and recorded the time taken to execute
each query. Figure~\ref{fig:point-queries} (a) illustrates that the
\textit{vector} implementation exhibits the poorest performance, which aligns
with expectations due to its linear search complexity. In Rocksdb, the point
queries are performed by first sorting the write buffer and then performing a
binary search for \textit{vector}. Notably, the graph's spikes can be
attributed to the overhead incurred by sorting the \textit{vector} write buffer
prior to performing a binary search for the point queries. The sort function of
the standard library of C++ uses an introspective sorting by doing a selection
of algorithms dynamically to optimize performance. It begins with quicksort,
switches to heapsort when the recursion depth exceeds a level based on the
number of elements begin sorted and it switches to insertion sort when the
number of elements is below some threshold. Figure~\ref{fig:point-queries} (b)
and (c) demonstrate that the \textit{hash skip list} (for \textit{X} 6 and
\textit{H} 100K) exhibits the best performance for point queries, while the
\textit{hash link list} performs better than the \textit{vector} and the
\textit{skip list}. Figure~\ref{fig:point-queries} (d), (e) and (f) illustrate
that the bad choice of prefix length and the number of buckets can lead to
suboptimal performance for point queries while using \textit{hash table} based
write buffer.

\Paragraph{Range Queries} irrespective of their selectivity span, exhibit distinct performance traits compared to point queries.
During a range query, the write buffer is tasked with locating all keys within the query bounds and returning them in an
ordered sequence. The \textit{vector}, due to its lack of inherent order, is less efficient in handling range queries,
as it necessitates a linear search to locate the keys within the range or to sort the keys and perform binary serach and then run sequential scan. In contrast, the \textit{skip list}
offers improved handling of both short and long range queries. It operates by initially identifying the key that is
greater than or equal to the lower bound of the range, then traverses the list in sequence until it reaches the upper
bound or encounters a greater key.

The \textit{hash skip list} and \textit{hash linked list} are designed to
optimize range queries by leveraging the $O(1)$ time complexity of a hash table
to locate the correct hash bucket. This performance comes with an extra
restriction which is that the whole range should be of the same prefix. The
range query flow will be as follows: $\left. 1\right)$ locate the correct hash
bucket, $\left. 2\right)$ perform a seek operation in the bucket to locate the
target or a key greater than target, $\left. 3\right)$ scan the bucket and
return the keys that are within the range. Note the keys are sorted in the
buckets.

The other scenario would be when the range query spans over multiple prefixes.
This can also happen if the prefix is large enough (lets say 3, the prefix
sample space will be $256^3$) and the number of buckets are less (lets say
10K). Now each bucket will have keys from on average $256^3/10K$ prefixes. In
this case, the \textit{hash skip list} and the \textit{hash linked list} will
perform a copy and sort operations to find all the keys within the range. It
has to iterate over all the buckets and perform seek operation to locate the
keys. In Rocksdb, it creates a new skiplist based view for all the keys within
the qualifying buckets and then return the keys that are within the range by
filtering the keys. The performance depends on the number of buckets that
qualify the range and then the number of keys it needs to seek over to filter
the keys. If the range query spans over multiple buckets and the number of keys
that qualify the range query is less then the range query will be more resource
intensive. It need to pull more number of buckets and then filter more number
of keys. If the range query spans over few buckets and the number of keys that
qualify the range query is more then the range query will be less resource
intensive as it needs to pull less number of buckets and then filter less
number of keys.

We ran our experiments with different selectivity where the start and end keys
are not similar and recorded the time taken to execute each range query.
Figure~\ref{fig:range-queries1} shows the comparison of range queries
performance for different write buffer implementations. The \textit{vector}
implementation shows the poorest performance for both short and long range
queries. The increase in the prefix length with the same number of buckets for
small range queries leads to the worst performance for \textit{hash skip list}
and \textit{hash linked list} whereas the increase in number of buckets with
the same prefix length shows better results.

\subsection{Writes Performance}

To measure the write performance of the write buffer implementations, we
conducted an experiment in which we inserted 140K entries, each 64 bytes in
size, and recorded the time taken to execute each insert query.
Figure~\ref{fig:insert-queries1} illustrates the comparison of write
performance across different write buffer implementations.

The ingestion process for a \textit{vector} write buffer is analogous to
appending bytes to an array. A typical vector implementation dynamically
allocates memory, doubling the allocation when the existing space is exhausted.
This can leads to performance degradation due to the repetitive memory copying
required during reallocation. The pre-allocation of memory serves as a remedy
to this inefficiency. As depicted in Figure~\ref{fig:insert-queries1} (a), the
dynamic memory allocation in a \textit{vector} leads to latency spikes during
ingestion. In contrast, Figure~\ref{fig:insert-queries1} (b) shows the enhanced
performance for a pre-allocated \textit{vector}, where the memory of $N \times
    (E+10) \approx $ 11 MB was pre-allocated to write buffer. The inherent sorting
nature of \textit{skip list} also slow down write operations as it requires
$O(\log n)$ time to locate the correct insertion location for every new key
whereas, a \textit{vector} simply append the new key to the end of the array
and shows a superior write performance over a \textit{skip list}.

Both the \textit{hash skip list} and the \textit{hash linked list} exhibit
improved write performance compared to the \textit{skip list}. These structures
capitalize on the $O(1)$ time complexity afforded by a hash table to identify
the correct hash bucket. This is followed by $O(\log \frac{P \times B}{H})$ for
searching a key within the \textit{hash skip list}, and on average $O(\frac{P
        \times B}{H})$ in the \textit{hash linked list}. The improved performance
(assuming uniform data distribution) can be achieved with a careful choice of
prefix length (\textit{X}) and bucket count (\textit{H}). The results of our
experiments, as plotted in Figure~\ref{fig:insert-queries1} (c) and (d),
demonstrate the superior performance of these hash-based structures. The
latency profiles over 4K insert queries indicate that the \textit{skip list} is
5 times and 738 times slower than the \textit{hash skip list} and the
\textit{hash linked list}, respectively.

\subsection{Flush \& Compaction}
The rate of flushes in LSM can also bump the overall write amplification and
impact the performance of the system. For every flush operation, the write
buffer is written to the disk as a new file. Every time the write buffer is
flushed to the disk, the system has two choices: $\left. 1\right)$ create a new
file and write to the level-0, or $\left. 2\right)$ merge the buffer with the
existing files on the level-0 and then write the merged file back to the
level-0. The first choice might lead to the large number of smaller files to
level-0, which will increase the number of files handle that a system has to
manage. The second choice will increase the write amplification as the system
has to read the existing files, merge them with the buffer and then write the
merged file back to the level.

The memory buffer like \textit{hash skip list} and \textit{hash linked list} with
large number of buckets can be full very quickly because of the hash table and
pointers overhead. This leads to the frequent flushes which will either
increase the number of files in level-0 or the write amplification of the
system. To measure the number of flushes and compactions, we ran our
experiements with unique 1 million inserts and recorded the number of flushes,
compactions and total number of bytes read and written to the disk.
