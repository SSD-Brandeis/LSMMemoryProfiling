\subsection{Experimental Setup}
To conduct these experiments, we used a Chameleon machine equipped with a configuration of 187 GiB of RAM and 
48 cores. We used the RocksDB~\cite{Facebook2021} storage engine to evaluate the performance of the memtable 
implementations. The RocksDB was configured to use only one thread for compactions and one thread for flushes to ensure
that the performance is not affected by the compaction and flush threads. We also disabled the write-ahead log (WAL)
to ensure that the performance is not affected by the WAL. Below are the parameters that we use across all the 
experiments:
\begin{itemize}[leftmargin=*,labelindent=0mm, itemsep=0.2\baselineskip]
    \item \textit{E} (entry size) = 64 bytes
    \item \textit{B} (number of entries that fit into a page) = 64
    \item \textit{P} (size of buffer in pages) = 4096
    \item \textit{T} (size ratio between adjacent levels) = 4
    \item \textit{M} (main memory allocated to write buffer) = 16 MB
\end{itemize}

\subsection{Memory Footprint}
The memory requirements of various memtable structures vary when tasked with storing the same volume of data. For 
instance, a \textit{skip list} generally requires more memory than a \textit{vector} due to its layered structure and 
the necessity to maintain pointers. Similarly, a \textit{hash skip list} has a higher memory footprint than a 
 \textit{skip list} because of the additional overhead from a hash table and pointers. This experiment seeks to reveal 
the trade-offs between the memory footprint and the storage capacity (number of entries) of each memtable 
implementation, providing insights into their efficiency and appropriateness for different scenarios.

We conducted the experiment by inserting 1 million entries into each memtable implementation and measured the memory 
footprint. The bucket count (\textit{bck}) and the prefix length (\textit{l}) for the \textit{hash skip list} and 
 \textit{hash link list} were set to 1 and 0, respectively. Figure~\ref{fig:memory_footprint} demonstrates that the 
\textit{vector} exhibits the lowest memory footprint due to its contiguous storage and absence of pointers, whereas 
the \textit{hash skip list} displays the highest memory footprint, reflecting its larger memory requirement. 
The \textit{skip list} and \textit{hash link list} have memory footprints that lie between the \textit{vector} and 
the  \textit{hash skip list}.

\subsection{Reads Performance}

\Paragraph{Point Queries} The performance of point queries is a critical factor in the selection of a memtable 
implementation. In the case of a \textit{vector}, point queries exhibit an $O(n)$ complexity, as the memtable must be 
scanned linearly to locate the required key. In contrast, \textit{skip list} implementations 
offer improved read performance, with point queries achieving $O(\log n)$ complexity. The  \textit{hash skip list} and \textit{hash skip list}, in 
particular, can deliver the best performance for point queries if the prefix length and the number of buckets are 
optimally tuned. It is noteworthy that the performance varies based on configuration, 
and it falls between \textit{vector} and \textit{skip list}.

The prefix length (\textit{l}) and the number of buckets (\textit{bck}) can fairly influence the performance of the
 \textit{hash skip list} and the  \textit{hash link list}. There are two extreme cases: $\left. 1\right)$ setting the prefix length too 
small, for example, to 0, results in all keys being stored in a single bucket, $\left. 2\right)$ increasing the prefix length to the 
maximum length of the keys while maintaining only one bucket leads to the same bottleneck. Ideally, the number of 
buckets should approximate $256^{\textit{l}}$ (assuming the character based prefix) to optimize performance. 
This strategy, however, poses a challenge as it 
necessitates a substantial increase in the number of buckets—approximately 65K for a \textit{l} of 2 and about 16M for 
a \textit{l} of 3—thereby significantly raising the memory overhead due to the need for each bucket to maintain a 
pointer to a  \textit{skip list}, unless it is empty. For point queries, achieving an $O(1)$ complexity is possible when 
the \textit{l} is adjusted to the maximum key length, coupled with $256^{\textit{l}}$ buckets, provided that sufficient 
memory is available. Moreover, the efficiency of point queries is further optimized when the distribution of keys across 
buckets is uniform. Thus, the \textit{hash skip list} can attain optimal performance for point queries through careful 
adjustment of the prefix length and the number of buckets.

The inherent increase in memory attributes to the layered structure of the  \textit{hash skip list}. 
The \textit{hash link list} emerges as a more memory-efficient alternative. Despite this, point queries can still 
maintain commendable performance provided that the buckets are uniformly filled and contains a smaller number of 
entries.

To assess the performance of point queries on Memtables, we conducted an experiment in which we inserted 140K entries, 
each 64 bytes in size, with random interleaved 200 point queries and recorded the time taken to execute each query. 
Figure~\ref{fig:point-queries} (a) illustrates that the \textit{vector} implementation exhibits the poorest performance, 
which aligns with expectations due to its linear search complexity. In Rocksdb, the point queries are performed by
first sorting the memtable and then performing a binary search for \textit{vector}. Notably, the graph's spikes can be attributed to 
the overhead incurred by sorting the \textit{vector} memtable prior to performing a binary search for the point queries. The 
sort function of the standard library of C++ uses different sorting algorithms to optimize performance.
Figure~\ref{fig:point-queries} (b) and (c) demonstrate that the \textit{hash skip
list} (for \textit{l} 6 and \textit{bck} 100K) exhibits the best performance for point queries, while the \textit{hash 
link list} performs better than the \textit{vector} and the \textit{skip list}. Figure~\ref{fig:point-queries} (d), (e) 
and (f) illustrate that the bad choice of prefix length and the number of buckets can lead to suboptimal performance
for point queries while using \textit{hash table} based memtable. 

\Paragraph{Range Queries} irrespective of their selectivity span, exhibit distinct performance traits compared to point queries.
During a range query, the memtable is tasked with locating all keys within the query's bounds and returning them in an 
ordered sequence. The \textit{vector}, due to its lack of inherent order, is less efficient in handling range queries,
as it necessitates a linear search to locate the keys within the range or to sort the keys. In contrast, the \textit{skip list}
offers improved handling of both short and long range queries. It operates by initially identifying the key that is 
greater than or equal to the lower bound of the range, then traverses the list in sequence until it reaches the upper 
bound or encounters a greater key. 

\inprogress{The \textit{hash skip list} and \textit{hash link list} are designed to optimize range queries by leveraging the
$O(1)$ time complexity of a hash table to locate the correct hash bucket. The range queries are then executed by
either searching the \textit{skip list} or the \textit{link list} for the target key. The performance of range queries
in these memtables is influenced by the prefix length and the number of buckets. The range queries will be faster
based on the prefix length and the number of buckets that are configured. If prefix length is small and the number of
buckets are less then the longer range queries will be faster as, for every prefix we will return more keys and we may need
to filter a small number the keys. If a larger prefix is choosen with small number of buckets then the smaller range 
queries will be faster as for every prefix we will return less keys and we might not need to filter the keys.}

\subsection{Writes Performance}

To measure the write performance of the memtable implementations, we conducted an experiment in which we inserted 140K 
entries, each 64 bytes in size, and recorded the time taken to execute each insert query. 
Figure~\ref{fig:insert-queries1} illustrates the comparison of write performance across different memtable 
implementations.

The ingestion process for a \textit{vector} memtable is analogous to appending bytes to an array. A typical vector 
implementation dynamically allocates memory, doubling the allocation when the existing space is exhausted. This can 
leads to performance degradation due to the repetitive memory copying required during reallocation. The pre-allocation of 
memory serves as a remedy to this inefficiency. As depicted in Figure~\ref{fig:insert-queries1} (a), the dynamic memory 
allocation in a \textit{vector} leads to latency spikes during ingestion. In contrast, Figure~\ref{fig:insert-queries1} 
(b) shows the enhanced performance for a pre-allocated \textit{vector}, where the memory of $N \times (E+10) \approx $ 
11 MB was pre-allocated to memtable. The inherent sorting nature of \textit{skip list} also slow down write operations 
as it requires $O(\log n)$ time to locate the correct insertion location for every new key whereas, a \textit{vector} 
simply append the new key to the end of the array and shows a superior write performance over a \textit{skip list}.

Both the \textit{hash skip list} and the \textit{hash link list} exhibit improved write performance compared to the 
\textit{skip list}. These structures capitalize on the $O(1)$ time complexity afforded by a hash table to identify the 
correct hash bucket. This is followed by $O(\log \frac{P \times B}{bck})$ for searching a key within the \textit{hash 
skip list}, and on average $O(\frac{P \times B}{bck})$ in the \textit{hash link list}. The improved performance (assuming uniform data distribution) 
can be achieved with a careful choice of prefix length (\textit{l}) and bucket count (\textit{bck}). The results of our 
experiments, as plotted in Figure~\ref{fig:insert-queries1} (c) and (d), demonstrate the superior performance of these 
hash-based structures. The latency profiles over 4K insert queries indicate that the \textit{skip list} is 5 times and 
738 times slower than the \textit{hash skip list} and the \textit{hash link list}, respectively.

\subsection{Flushes \& Compactions}
The rate of flushes in LSM can also bump the overall write amplification and impact the performance of the system. For every 
flush operation, the memtable is written to the disk as a new SSTable. Everytime the write buffer is flused to the disk,
the system has two choices: $\left. 1\right)$ create a new SSTable and write to the level-0, or $\left. 2\right)$ merge
the buffer with the existing SSTables on the level-0 and then write the merged SSTable back to the level-0. The first
choice might lead to the large number of smaller files to level-0, which will increase the number of files handle
that a system has to manage. The second choice will increase the write amplification as the system has to read the
existing SSTables, merge them with the buffer and then write the merged SSTable back to the level.

The memetable like \textit{hash skip list} and \textit{hash link list} with large number of buckets can be full
very quickly because of the hash table and pointers overhead. This leads to the frequent flushes which will
