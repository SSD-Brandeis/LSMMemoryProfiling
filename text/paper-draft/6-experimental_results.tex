\subsection{Experimental Setup}
To conduct these experiments, we use a virtual machine equipped with a
configuration of 187GB of RAM and 48 cores. 
We use the RocksDB~\cite{Facebook2021} storage engine to evaluate the performance of the write buffer implementations. 
RocksDB is configured to use only one thread for compactions and one thread for flushes to ensure that the performance is not affected by the compaction and flush threads. 
We also disable the write-ahead log (WAL) to ensure that the performance is not affected by the WAL. 
Unless otherwise mentioned, the size of a key-value pair is $64$B and the size of each disk page is $4$KB. 
Size of the memory buffer is $16$MB. 
The LSM-tree has a size ratio of $4$, i.e., each level on disk is $4\times$ larger than the previous level.


\subsection{Memory Footprint}
The memory requirements of various write buffer structures vary when tasked
with storing the same volume of data. For instance, a \textit{skip-list}
generally requires more memory than a \textit{vector} due to its layered
structure and the necessity to maintain pointers. Similarly, a \textit{hash
    skip-list} has a higher memory footprint than a \textit{skip-list} because of
the additional overhead from a hash table and pointers. This experiment seeks
to reveal the trade-offs between the memory footprint and the storage capacity
(number of entries) of each write buffer implementation, providing insights
into their efficiency and appropriateness for different scenarios.

We conducted the experiment by inserting 1.5M entries into each write
buffer implementation and measured the memory footprint. The bucket count
(\textit{H}) and the prefix length (\textit{X}) for the \textit{hash skip-list}
and \textit{hash linked-list} varied from 10 to 1000 and 2 to 10, respectively.
Figure~\ref{fig:memory_footprint} demonstrates that the \textit{vector}
exhibits the lowest memory footprint due to its contiguous storage and absence of pointers (hence used more space to store the data), whereas the \textit{hash skip-list} displays the highest memory
footprint, reflecting its larger memory requirement. The \textit{skip-list} and
\textit{hash linked-list} have initial memory footprints close to \textit{skip-list} for small prefix and bucket counts but it increases with the increase in the prefix length (Figure~\ref{fig:memory_footprint} (a)) and the number of buckets (Figure~\ref{fig:memory_footprint} (b)).

\subsection{Read Performance}

\Paragraph{Point Queries} 
An operation that retrieves a single key from the write buffer (or data store) is referred as a point query.
The performance of point queries plays a crucial role in the selection of a write buffer implementation.
In the case of a \textit{vector}, point queries exhibit an $O(n)$ (where $n$ is the number of entries in the buffer) complexity, as the buffer must be scanned linearly to locate the required key.
An alternative approach is to sort the write buffer and perform a binary search, which reduces the search time to $O(\log n)$ but adds an overhead of $O(n \log n)$ for sorting.
In contrast, \textit{skip-list} implementations offer improved read performance, with point queries achieving $O(\log n)$ complexity. 
The \textit{hash skip-list} and \textit{hash skip-list}, in particular, can deliver the best performance for point queries if the prefix length and the number of buckets are optimally tuned. 
It is noteworthy that the performance varies based on configuration, and it falls between \textit{vector} and \textit{skip-list} in the normal cases.


The prefix length (\textit{X}) and the number of buckets (\textit{H}) can fairly influence the performance of the \textit{hash skip-list} and the \textit{hash linked-list}. 
There are two extreme cases: $\left. 1\right)$ setting the prefix length too small, for example, to 0, results in all keys being stored in a single bucket, $\left. 2\right)$ increasing the prefix length to the maximum length of the keys while maintaining only one bucket leads to the same bottleneck. 
Ideally, the number of buckets should approximate $256^{\textit{X}}$ (assuming the character based prefix) to optimize performance. 
This strategy, however, poses a challenge as it necessitates a substantial increase in the number of buckets — approximately 65K for a \textit{X} of 2 and about 16M for a \textit{X} of 3 — thereby significantly raising the memory overhead due to the need for the pointer that needs to be maintained for each bucket, unless it is empty.
For point queries, achieving an $O(1)$ complexity is possible when the \textit{X} is adjusted to the maximum key length, coupled with $256^{\textit{X}}$ buckets, provided that sufficient memory is available. 
Moreover, the efficiency of point queries is further optimized when the distribution of keys across buckets is uniform.
Thus, the \textit{hash skip-list} can attain optimal performance for point queries through careful adjustment of the prefix length and the number of buckets. 
Due to the layered structure of the \textit{hash skip-list}, the memory overhead can be significantly higher.
The \textit{hash linked-list} emerges as a more memory-efficient alternative. 
Despite this, point queries can still maintain commendable performance provided that the buckets are uniformly filled and contains a smaller number of entries.



\Paragraph{Point Queries Experiments} 
To assess the performance of point queries on write buffers, we conducted our experiments in which we inserted 140K entries, each 64 bytes in size, with random interleaved 200 point queries and recorded the time taken to execute each query. 
Figure~\ref{fig:point-queries} (a) illustrates that the \textit{vector} implementation exhibits the poorest performance, which aligns with expectations due to its linear search complexity. 
In RocksDB, the point queries are performed by first sorting the write buffer and then performing a binary search for \textit{vector}. 
Notably, the graph's spikes can be attributed to the overhead incurred by sorting of \textit{vector} buffer prior to performing a binary search for the point queries. 
The sort function of the standard library of C++ uses an introspective sorting by doing a selection of algorithms dynamically to optimize performance. 
It begins with quicksort, switches to heapsort when the recursion depth exceeds a level based on the number of elements begin sorted and it switches to insertion sort when the number of elements is below some threshold. 
Figure~\ref{fig:point-queries} (b) and (c) demonstrate that the \textit{hash skip-list} (for \textit{X} 6 and \textit{H} 100K) exhibits the best performance for point queries, while the \textit{hash linked-list} performs better than the \textit{vector} and the \textit{skip-list}. 
Figure~\ref{fig:point-queries} (d), (e) and (f) illustrate that the bad choice of prefix length and the number of buckets can lead to suboptimal performance for point queries while using \textit{hash table}-based write buffer.

\Paragraph{Range Queries} 
This operation is used to scan a range of keys from the write buffer, which starts from the lower bound and return all keys till the upper bound.
Irrespective of their selectivity span, range queries exhibit distinct performance traits compared to point queries.
During a range query, the write buffer is tasked with locating all keys within the query bounds and returning them in an ordered sequence. 
The \textit{vector}, due to its lack of inherent order, is less efficient in handling range queries, as it necessitates a linear search to locate the keys within the range, alternatively it can also sort the keys and perform binary search for lower bound, and then run sequential scan. 
In contrast, the \textit{skip-list} offers improved handling of both short and long range queries. 
% It operates by initially identifying the key that is greater than or equal to the lower bound of the range, then traverses the list in sequence until it reaches the upper bound or encounters a greater key.

\Paragraph{Prefix-based Filtering} 
The \textit{hash skip-list} and \textit{hash linked-list} are designed to optimize range queries by leveraging the $O(1)$ time complexity of a hash table to locate the correct hash bucket. 
This performance comes with an extra restriction which is that the whole range should be of the same prefix. 
The range query flow will be as follows: $\left. 1\right)$ locate the correct hash bucket, $\left. 2\right)$ perform a seek operation in the bucket to locate the target or a key greater than target, $\left. 3\right)$ scan the bucket and return the keys that are within the range. 
Note the keys are sorted in the buckets.

The another scenario can be when the range query spans over multiple prefixes.
This can also happen if the prefix is large enough (lets say 3, the prefix
sample space will be $256^3$) and the number of buckets are less (lets say
10K). Now each bucket will have keys from on average $256^3/10K$ prefixes. In
this case, the \textit{hash skip-list} and the \textit{hash linked-list} will
perform a copy and sort operations to find all the keys within the range. It
has to iterate over all the buckets and perform seek operation to locate the
keys. In RocksDB, it creates a new skip-list based view for all the keys within
the qualifying buckets and then return the keys that are within the range by
filtering the keys. The performance depends on the number of buckets that
qualify the range and then the number of keys it needs to seek over to filter
the keys. If the range query spans over multiple buckets and the number of keys
that qualify the range query is less then the range query will be more resource
intensive. It need to pull more number of buckets and then filter more number
of keys. If the range query spans over few buckets and the number of keys that
qualify the range query is more then the range query will be less resource
intensive as it needs to pull less number of buckets and then filter less
number of keys.

\subsection{Write Performance}

To measure the write performance of the write buffer implementations, we
conducted an experiment in which we inserted 140K entries, each 64 bytes in
size, and recorded the time taken to execute each insert query.
Figure~\ref{fig:motivation} (b) illustrates the comparison of write
performance across different write buffer implementations.

\Paragraph{Ingestion Experiments} The ingestion process for a \textit{vector} write buffer is analogous to
appending bytes to an array. A typical vector implementation dynamically
allocates memory, doubling the allocation when the existing space is exhausted.
This can leads to performance degradation due to the repetitive memory copying
required during reallocation. The pre-allocation of memory serves as a remedy
to this inefficiency. As depicted in Figure~\ref{fig:motivation} (b), the
dynamic memory allocation in a \textit{vector} leads to latency spikes during
ingestion. In contrast, Figure~\ref{fig:motivation} shows the enhanced
performance for a pre-allocated \textit{vector}, where the memory of $N \times
    (E+10) \approx $ 11 MB was pre-allocated to write buffer. The inherent sorting
nature of \textit{skip-list} also slow down write operations as it requires
$O(\log n)$ time to locate the correct insertion location for every new key
whereas, a \textit{vector} simply append the new key to the end of the array
and shows a superior write performance over a \textit{skip-list}.

Both the \textit{hash skip-list} and the \textit{hash linked-list} exhibit
improved write performance compared to the \textit{skip-list}. These structures
capitalize on the $O(1)$ time complexity afforded by a hash table to identify
the correct hash bucket. This is followed by $O(\log \frac{P \times B}{H})$ for
searching a key within the \textit{hash skip-list}, and on average $O(\frac{P
        \times B}{H})$ in the \textit{hash linked-list}. The improved performance
(assuming uniform data distribution) can be achieved with a careful choice of
\textit{X} and \textit{H}. The results of our
experiments, as plotted in Figure~\ref{fig:motivation} does not shows the insert latency. 
However, the latency for inserting 140K records indicate that the \textit{skip-list} is
5 times and 7 times slower than the \textit{hash skip-list} and the
\textit{hash linked-list}, respectively.

\subsection{Flush \& Compaction}
\Paragraph{Compactions \& Flushes Impact} The rate of flushes in LSM-tree can increase the read and write amplifications, which can impact the overall performance of the system. 
For every flush operation, the write buffer is written to the disk as a new file. 
The system have two choices: either $\left. 1\right)$ to create a new file and write to the the first level on disk, i.e., to Level 0, or $\left. 2\right)$ read the existing (overlapping) files from the Level 0 and then write the merged file back to the Level 0.
The first choice may lead to the more number of smaller files at Level 0, which will increase the number of file handles that needs to be managed by the system.
The second choice will increase the read and write amplifications as the system needs to read the existing files, merge them with the buffer and then write the merged file(s) back to the level.

The memory buffer like \textit{hash skip-list} and \textit{hash linked-list} with large number of buckets can get full very quickly because of the hash table and pointers overhead. 
This leads to the frequent flushes which will end up in one of the above discussed scenarios. 
To measure the number of flushes and compactions, we run our experiments with unique 1M inserts and recorded the number of flushes, compactions and total number of bytes read and written to the disk.
Figure~\ref{fig:flush-compaction} shows the comparison of flushes and compactions for different write buffer implementations. The \textit{hash skip-list} due to its large memory requirement shows almost $0.5\times$ more 
flushes and $0.7\times$ more writes. The \textit{vector} implementation shows the least number of flushes and compactions among all the write buffer implementations.
