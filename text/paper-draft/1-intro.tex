\Paragraph{LSM-Based Key-Value Stores} 
Log-structured merge (LSM) trees are widely used in the storage engines of modern NoSQL key-value stores~\cite{Idreos2020,Luo2020b,ONeil1996}. 
LSM-tree is a highly ingestion-optimized, out-of-place data structure that stores data on disk as a hierarchical collection of \textit{immutable sorted runs}~\cite{Sarkar2022b,Sarkar2022a}. 
Immutability in the file structure means updates and deletes in LSM-based storage engines are always realized out of place, by inserting new entries (or meta-entries, in case of deletes) that logically invalidate the older target entries~\cite{Sarkar2020,Sarkar2023a}. 
By avoiding in-place updates and deletes LSM-engines are able to offer superior ingestion performance. 
To facilitate fast queries, commercial LSM-engines often employ auxiliary data structures, such as Bloom filters and fence pointers, that reduces the overall number of accesses to slower disks. 
Due to these advantages, a large number of commercial NoSQL systems, including LevelDB~\cite{GoogleLevelDB} and BigTable~\cite{Chang2006} at Google, RocksDB~\cite{FacebookMyRocks} at Meta, X-Engine~\cite{Huang2019} at Alibaba, WiredTiger~\cite{WiredTiger} at MongoDB, CockroachDB~\cite{CockroachDB2021} at Cockroach Labs, and AsterixDB~\cite{Alsubaiee2014}, Cassandra~\cite{ApacheCassandra}, HBase~\cite{ApacheHBase}, and Accumulo~\cite{ApacheAccumulo} at Apache, have adopted LSM-trees to implement their storage layer.

\Paragraph{LSM-Buffer and its Implementation} 
To ensure high throughput for inserts and to avoid writing data eagerly to slower secondary storage, LSM-trees buffer the incoming entries first in memory~\cite{Luo2019c,Sarkar2022a}. 
The buffer (also called the \textit{memtable}) in the main memory acts as the primary storage component of an LSM-tree~\cite{ONeil1996}. 
The buffer has a predetermined capacity (typically, 1MB-128MB)~\cite{Cao2020,Dong2017}. 
Once full, the entries in the buffer are sorted on the key and are written to the first level of the hierarchical collection of immutable sorted runs on disk. 
We refer to this process of persisting the contents of the buffer to disk as \textit{flush}~\cite{Sarkar2022b}. 
Buffering the inserts in memory allows LSM-engines to amortize the number of disk I/Os performed while inserting data which, by design, makes them optimized for ingestion~\cite{Sarkar2021c}. 
% \Paragraph{Buffer Implementation and its Performance Implications}
In practice, systems use different data structures with widely varying implementations to design the memory buffer in LSM-based storage engines. 
\red{Vector, skip-list, linked-list, and hybrid data structures, such as hash skip-list and hash linked-list, are some of the commonly used data structures in commercial LSM-based storage engines~\cite{Dong2021a,Facebook2021}.} 
Each data structure brings a different tradeoff between performance and memory footprint~\cite{Athanassoulis2016}.

\Paragraph{Challenge: Determining the ``Best'' Buffer Implementation} 
We point out that despite being a disk-based data structure, \textbf{the choice of data structure has huge implications on the overall performance} of an LSM-engine. 
We highlight two general scenarios where the choice of the buffer data structure and its implementation has critical performance implications. 
(A) For the same buffer data structure, the performance numbers may vary significantly with the \textit{slightest shift in the workload characteristics}. 
In Figure~\ref{fig:motivation1}, we show that the ingestion throughput of an LSM-buffer, implemented as a (static) vector drop by $X\%$ if the input workload shifts from insert-only to having only $0.1\%$ point queries (and $99.9\%$ inserts). 
(B) For the same data structure, \textit{different implementations and tunings} also has a significant impact on performance. 
For example, as shown in Figure~\ref{fig:motivation1}, the tail latency of a dynamically allocated vector can be up to \red{two orders of magnitude} higher than that of a pre-allocated vector. 
Similarly, the point number of hash-buckets in a hash linked-list or a hash skip-list may affect the ingestion throughput significantly.

\Paragraph{Solution}
The prior information of the workload characteristics with a better understanding of the underlying data structures performance can help in making better decisions while choosing the write buffer for a particular workload.
The main aim of this paper is to elucidate the role of write buffers within Log-structured Merge Trees (LSM), examine how various implementations affect memory usage and performance, and provide a framework for selecting the optimal write buffer based on specific workloads and available resources.
The contributions of our work is as follows.

\begin{enumerate}[leftmargin=*,labelindent=0mm, itemsep=0.2\baselineskip]
    \item \textbf{Memory Analysis}: A thorough examination of the memory footprint across different write buffer implementations, including \textit{skip list, vector, hash skip list, and hash linked list}.
    \item \textbf{Performance Modeling}: We develop a performance model for write buffer implementations that details the time complexity of read and write operations. 
    This model facilitates a comparative analysis of the performance of various write buffer types under differing workload conditions.
    \item \textbf{Buffer Selection Insights}: Based on our performance analysis and modeling, we offer recommendations for choosing the most appropriate write buffer implementation, taking into account the specific demands of the workload and the resources at hand.
\end{enumerate}
These contributions aim to enhance understanding and decision-making regarding write buffer configurations within LSM-trees, addressing both theoretical and practical considerations in database management and optimization. 
To underscore the real-world applicability of our research, we employ RocksDB—a widely recognized key-value store—as a case study. 
This choice allows us to validate our findings in a practical setting, demonstrating their relevance and utility in optimizing data storage solutions.
We will try to answer the questions like,

    \item What is the right buffer for a insert-only workload?
    \item Why can we have write stalls while using vector write buffer?
    \item In what scenarios we use hash skip list or linked list buffer?
    \item On what paramerter default choice can be worse than others?
\end{enumerate}
