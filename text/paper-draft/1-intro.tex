\Paragraph{LSM-Based Key-Value Stores} 
Log-structured merge (LSM) trees are widely used in the storage engines of modern NoSQL key-value stores~\cite{Idreos2020,Luo2020b,ONeil1996}. 
LSM-tree is a highly ingestion-optimized, out-of-place data structure that stores data on disk as a hierarchical collection of \textit{immutable sorted runs}~\cite{Sarkar2022b,Sarkar2022a}. 
Immutability in the file structure means updates and deletes in LSM-based storage engines are always realized out of place, by inserting new entries (or meta-entries, in case of deletes) that logically invalidate the older target entries~\cite{Sarkar2020,Sarkar2023a}. 
By avoiding in-place updates and deletes LSM-engines are able to offer superior ingestion performance. 
To facilitate fast queries, commercial LSM-engines often employ auxiliary data structures, such as Bloom filters and fence pointers, that reduces the overall number of accesses to slower disks. 
Due to these advantages, a large number of commercial NoSQL systems, including LevelDB~\cite{GoogleLevelDB} and BigTable~\cite{Chang2006} at Google, RocksDB~\cite{FacebookMyRocks} at Meta, X-Engine~\cite{Huang2019} at Alibaba, WiredTiger~\cite{WiredTiger} at MongoDB, CockroachDB~\cite{CockroachDB2021} at Cockroach Labs, and AsterixDB~\cite{Alsubaiee2014}, Cassandra~\cite{apApacheCassandra}, HBase~\cite{ApacheHBase}, and Accumulo~\cite{ApacheAccumulo} at Apache, have adopted LSM-trees to implement their storage layer.

\Paragraph{Buffering Inserts} 
To ensure high throughput for inserts and avoid writing data eagerly to slower secondary storage, LSM-trees buffers the incoming entries first in memory~\cite{Luo2019c,Sarkar2022a}. 
The memory buffer (also often referred to as the \textit{memtable}) acts as the primary storage component of the LSM-tree~\cite{ONeil1996}. 
The buffer has a predetermined capacity and may be implemented as any data structure~\cite{Facebook2021}. 
Once the buffer is full, the entries are sorted based on the key and are written to the first level of the hierarchical collection of immutable sorted runs on disk. 
We refer to this process of persisting the contents of the buffer to the disk as \textit{flush}~\cite{Sarkar2021c}.

\Paragraph{Buffer selection}
The buffer selection is always overlooked while using LSM-based data stores. The \textit{skip list}~\cite{Pugh1990}, default buffer
in RocksDB~\cite{Facebook2021} and LevelDB, is not always the best choice for all types of workloads. This
is a probabilistic data structure that provides $O(\log n)$ time complexity for lookups and inserts, whereas, on the memory front, because of the layered structure and the necessity
to maintain pointers for each element, it requires extra memory than the actual data is stored.
Let us call this metadata for the data structure being used. More meta data means less number of entries that can fit
into the buffer and, hence more files to be flushed to the disk. It is a trade-off to facilitate the 
$O(\log n)$ time complexity for lookups and inserts. The \textit{vector}, an another implementation, provides
$O(1)$ time complexity for inserts with less metadata overhead as it simply writes at the end of a contiguous array of memory. However, 
\textit{vector} is not good for lookups and also requires more CPU resources to \textit{flush}~\cite{Sarkar2021c} the buffer to the disk as it is not sorted.

analysis across various scenarios or the lack of a documentation for write buffer implementations in the LSM-tree. The

\Paragraph{Solution}
The prior information of the worklaod characteristics with a better understanding of the underlying data structures performance can help in making better decisions while choosing the write buffer for a particular workload.
The main aim of this paper is to elucidate the role of write buffers within Log-structured Merge Trees (LSM), examine how various implementations affect memory usage and performance, and provide a framework for selecting the optimal write buffer based on specific workloads and available resources.
The contributions of our work is as follows.

\begin{enumerate}[leftmargin=*,labelindent=0mm, itemsep=0.2\baselineskip]
    \item \textbf{Memory Analysis}: A thorough examination of the memory footprint across different write buffer implementations, including \textit{skip list, vector, hash skip list, and hash linked list}.
    \item \textbf{Performance Modeling}: We develop a performance model for write buffer implementations that details the time complexity of read and write operations. 
    This model facilitates a comparative analysis of the performance of various write buffer types under differing workload conditions.
    \item \textbf{Buffer Selection Insights}: Based on our performance analysis and modeling, we offer recommendations for choosing the most appropriate write buffer implementation, taking into account the specific demands of the workload and the resources at hand.
\end{enumerate}
These contributions aim to enhance understanding and decision-making regarding write buffer configurations within LSM-trees, addressing both theoretical and practical considerations in database management and optimization. 
To underscore the real-world applicability of our research, we employ RocksDB—a widely recognized key-value store—as a case study. 
This choice allows us to validate our findings in a practical setting, demonstrating their relevance and utility in optimizing data storage solutions.
We will try to answer the questions like, 
\begin{enumerate}[label=(\alph*),leftmargin=*,labelindent=0mm, itemsep=0.2\baselineskip]
    \item What is the right buffer for a insert-only workload?
    \item Why can we have write stalls while using vector write buffer?
    \item In what scenarios we use hash skip list or linked list buffer?
    \item On what paramerter default choice can be worse than others?
\end{enumerate}
